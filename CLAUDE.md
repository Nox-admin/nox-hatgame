This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

iOS word-guessing party game ("Шляпа" / HatGame). Swift + SwiftUI + MVVM + SwiftData. iOS 17+, Swift 5.9. No external dependencies. Fully offline, pass-the-phone local multiplayer. Localized in ru / en / zh-Hans (RU is primary).

## Build & run

The Xcode project lives in `src/`. It is generated from `src/project.yml` via XcodeGen. `HatGame.xcodeproj` is **gitignored** — regenerate it after cloning or after changing `project.yml`:

```bash
cd src && xcodegen generate
```

```bash
# Build (CLI)
cd src
xcodebuild -project HatGame.xcodeproj -scheme HatGame \
  -destination 'platform=iOS Simulator,name=iPhone 15' build

# Run UI tests (the only test target)
xcodebuild -project HatGame.xcodeproj -scheme HatGame \
  -destination 'platform=iOS Simulator,name=iPhone 15' test

# Run a single UI test
xcodebuild ... test -only-testing:HatGameUITests/HatGameUITests/<testName>
```

There is no Makefile, Fastlane, lint config, or unit test target — only `HatGameUITests` (XCUITest).

## Architecture

MVVM with a single navigation coordinator and per-mode game ViewModels. The shape is unusual enough to read before editing.

### Navigation coordinator pattern

`GameViewModel` (`ViewModels/GameViewModel.swift`) is **not** a gameplay VM. It is a `@MainActor` navigation coordinator that:
- Owns `navigationState: NavigationState` (enum in `ContentView.swift`) — the entire app routes through a single `switch` in `ContentView.mainContent`. There is no `NavigationStack` push/pop for game screens.
- Lazily creates one of three mode-specific VMs on `startGame(with: GameConfig)`: `pairsViewModel`, `teamsViewModel`, `freeForAllViewModel`. Only one is non-nil at a time.
- Pipes the active child VM's `objectWillChange` into its own (BUG-022 fix) so `ContentView` re-renders when child state changes.
- Deliberately does **not** conform to `PausableViewModel` (TASK-006) — it has no timer; pause lives in mode VMs.

Each gameplay view observes its mode VM's `phase`/`navigationPhase` via `.onChange` and calls `gameViewModel.navigateTo(...)` to advance. Do not bypass this — adding direct `NavigationLink`s will break state.

### Three game modes

The three modes (`pairs`, `teams`, `freeForAll`) each have their own VM under `ViewModels/` and their own set of views (`PairsWaitingView`, `PairsGameplayView`, `PairsRoundEndView`, `PairsCircleEndView`, `TeamsWaitingView`, ..., `FreeForAllWaitingView`, ...). They all funnel into the shared `UnifiedFinalView` at game over. Mode-specific VMs share the `PausableViewModel` protocol for pause/resume.

### Game engine

`Services/GameEngine.swift` is the rules state machine (`.setup → .playing → .roundEnd → ... → .gameOver`). It owns `GameSession` and a `TimerService`. Critical invariant from BUG-001: **never recreate `timerService`** on round start — `@ObservedObject` subscriptions in views would drop. Reuse and call `start(duration:)`.

`DeckService.loadDeck(level:count:)` loads from `Resources/words.json` (RU primary), `words_en.json`, `words_zh.json`. Word selection is randomized per game (TASK-027); if fewer words exist than requested `hatSize`, it returns all available.

### Persistence

SwiftData is used minimally. `HatGameApp` registers a single `@Model` entity, `GameSettingsEntity` (BUG-008: `GameSettings` itself is a plain struct; only the persisted form is a `@Model`). Don't add `@Model` to gameplay structs.

### Localization

- Strings live in `Resources/Localizable.xcstrings` (canonical) plus `*.lproj/Localizable.strings` mirrors for ru/en/zh-Hans.
- All UI strings go through `Utils/L10n.swift`, which exposes nested enums (`L10n.Home.title`, etc.) backed by `"key".localized`. Properties are **computed**, not stored, so they react when `LanguageManager` changes language.
- `HatGameApp` sets `.id(languageManager.currentLanguage.rawValue)` on `ContentView` — switching language rebuilds the entire view tree to force re-reading L10n. Do not remove this `.id`.
- When adding a string: add it to `Localizable.xcstrings` for all three languages, then expose via `L10n`. Do not hardcode user-facing strings (TASK-081 cleaned these up).
